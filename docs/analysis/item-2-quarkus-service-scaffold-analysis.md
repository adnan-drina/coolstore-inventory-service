# Item 2 Analysis: Quarkus Service Scaffold

## Scope

Add the first working `coolstore-inventory-service` Quarkus scaffold at the repository root. Keep the implementation bounded to inventory read APIs, deterministic tests, health, and metrics. Do not add PostgreSQL runtime wiring, Tekton assets, GitOps manifests, or the reservation endpoint in this item.

## Findings

- Current Red Hat documentation identifies `3.27.3.SP1-redhat-00002` as the latest Red Hat build of Quarkus 3.27.x platform version.
- Red Hat build of Quarkus 3.27.x supports Java 21 on Red Hat Enterprise Linux and Red Hat OpenShift Container Platform.
- PostgreSQL is a tested integration for Red Hat build of Quarkus 3.27.x, but the plan for this item explicitly defers PostgreSQL runtime dependency and OpenShift catalog database wiring.
- The local machine has Java 21 and Maven 3.9.10, which is sufficient for local scaffold validation.
- The Red Hat build of Quarkus artifacts require the Red Hat Maven repository. A project-local Maven settings file is appropriate for this demo because it avoids assuming that every developer has a global Maven settings file.

## Implementation Decision

Use Red Hat build of Quarkus `3.27.3.SP1-redhat-00002` with Java 21 and Maven.

Add:

- `pom.xml` at the repository root;
- project-local `.mvn/settings.xml` and `.mvn/maven.config`;
- Maven wrapper files generated by Maven wrapper tooling;
- `com.redhat.coolstore.inventory` source files;
- deterministic REST tests for the three initial GET APIs;
- minimal `application.properties`;
- README, plan, catalog, and AGENTS updates reflecting that the scaffold now exists.

The initial service uses an in-memory repository so tests run without Docker, Dev Services, PostgreSQL, OpenShift, or a live cluster.

## Risks

- The Maven wrapper may need network access on first use to download Maven if it is not already cached.
- The service uses in-memory data only; persistence and OpenShift PostgreSQL behavior must not be claimed until a later item adds it.
- `catalog-info.yaml` remains preliminary until the Developer Hub metadata item.
- The legacy Python exercises still exist and should be archived only in a later cleanup item after this scaffold is reviewed.
